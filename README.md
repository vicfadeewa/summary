**План автоматизации тестирования сценария перехода к форме записи и заполнения формы «Инженер по тестированию» на сайте Нетологии**

**1. Перечень автоматизируемых сценариев**
**1.1. Сценарии перехода к форме записи:**

- переход на страницу профессии через меню «Каталог курсов» → «Программирование» → выбор профессии;
- переход на страницу профессии через меню «Каталог курсов» → «Полный каталог» → поиск по названию;
- переход на страницу профессии через раздел «Направления обучения» → «Программирование» → поиск по названию;
- переход к форме записи через кнопку «Записаться» вверху страницы профессии;
- переход к форме записи через прокрутку страницы до блока с формой.

**1.2. Сценарии заполнения формы записи (позитивные тесты):**

- все поля заполнены валидными данными (имя, телефон, e-mail), оба чек-бокса отмечены;
- заполнение формы авторизованным пользователем;
- проверка отображения сообщения об успешной отправки заявки
- перенаправление на страницу подтверждения

**1.3. Сценарии заполнения формы записи (негативные тесты):**

- пустое поле «Имя»;
- имя из 1 символа;
- имя с цифрами/спецсимволами;
- некорректный формат телефона (менее 11 цифр, буквы);
- некорректный e-mail (без @, неверная доменная часть);
- отправка формы с одним незаполненным полем;
- отправка формы с невалидными данными во всех полях;
- попытка отправки формы при неотмеченных чек-боксов;
- проверка сообщений об ошибках.


**2. Перечень используемых инструментов**

**2.1. IntelliJ IDEA**
**Назначение:** IDE для разработки и отладки тестов.
- поддержка Maven/Gradle, JUnit/TestNG, Git;
- удобный интерфейс для написания и запуска тестов;
- отладчик для пошагового анализа работы;
- поддержка генерации отчётов.

**2.2. Selenide**
**Назначение:** автоматизация UI-тестов для веб-приложений, упрощение взаимодействия с веб-интерфейсом.
- подходит для Java-проектов, упрощает написание читаемого и поддерживаемого кода;
- построен на базе Selenium WebDriver, но предлагает более лаконичный синтаксис и встроенные механизмы ожидания элементов.

**2.3. JUnit**
**Назначение:** написание и организация тестовых сценариев, параметризация тестов.
- JUnit 5 — современный фреймворк с модульной архитектурой, поддерживает динамические и параметризованные тесты;
- интегрируются с Selenide и Allure.

**2.4. Maven/Gradle**
**Назначение:** управление зависимостями проекта, сборка тестового кода.
- Maven обеспечивает стандартизированную структуру проекта, что упрощает совместную работу;
- Gradle предлагает большую гибкость, поддерживает инкрементальную сборку и кэширование, что ускоряет процесс в крупных проектах
- интеграция с CI/CD

**2.5. Git**
**Назначение:** контроль версий, совместная работа.
- поддержка ветвления и слияния упрощает параллельную разработку и управление изменениями
- интеграция с GitHub Actions

**2.6. Allure**
**Назначение:** генерация наглядных отчётов, управление тестовыми кейсами.
- предоставляет панель для отслеживания прогресса тестирования, фильтрация по статусам
- визуализация шагов теста и скриншоты


**3. Перечень необходимых разрешений, данных и доступов**

**3.1. Доступ к сайту Нетологии**
**Обоснование:** тестирование в продуктивной среде для реалистичных результатов

**3.2. Тестовые учётные записи и набор тестовых данных**
**Обоснование:** покрытие позитивных и негативных сценариев.
- авторизованные пользователи;
- неавторизованные пользователи (для проверки формы записи без входа в систему).
- валидные данные (корректные имена, телефоны, email);

**3.3. Доступ к документации**
**Обоснование:** точность тестирования и валидация.
- спецификации формы (обязательные поля, правила валидации);
- тексты чек‑боксов и ссылок на документы;
- описание бизнес‑логики (условия обязательной отметки чек‑боксов)


**4. Перечень и описание возможных рисков при автоматизации**

**4.1.  Изменение UI/UX сайта**
- перестановка и переименование элементов, смена классов/ID, редизайн

**4.2. Ограниченный доступ к среде**
- запрет на тестирование в продуктиве или отсутствие тестовой среды
- невозможность проверить сценарии в реальных условиях

**4.3. Неявная обязательность чек‑боксов**
- чек‑бокс может быть обязательным, но без явного маркера

**4.4. Капча и защита от ботов**
- форма может быть защищена механизмами, блокирующими автоматизированные запросы
(возможно использование API‑методов для отправки данных)

**4.5. Изменения в бизнес‑логике**
- обновление правил валидации полей или условий отправки формы
- тесты перестают соответствовать актуальным требованиям

**5. Перечень необходимых специалистов для автоматизации**

**5.1. Инженер по тестированию**
- разработка и поддержка автоматизированных тестов;
- настройка тестового окружения и интеграционных компонентов;
- анализ результатов тестирования и формирование отчётов;
- отладка сценариев при изменении UI/UX;
- ведение документации по тестовым кейсам.

**5.2. Аналитик или менеджер по проекту**
- сбор и уточнение требований к форме записи;
- валидация текстов подписей, чек‑боксов и юридических ссылок;
- согласование обязательных полей и правил валидации с командой разработки;
- коммуникация с заказчиком (Нетология) по изменениям в бизнес‑логике;
- контроль сроков выполнения этапов.

**5.3. Разработчик**
- решение технических проблем

**5.4. DevOps‑специалист или администратор сайта**
- развёртывание тестовой инфраструктуры (локальной или облачной);
- настройка доступа к продуктивной среде с учётом ограничений безопасности;
- обеспечение изоляции тестовых данных (предотвращение попадания в продакшн);
- мониторинг нагрузки на сервер при запуске тестов

- выявление обязательных/опциональных полей, логики чек‑боксов;
- составление матрицы сценариев (позитивные/негативные/граничные случаи);

Риски: изменение требований

- инициализация репозитория, настройка .gitignore

- автоматизация переходов к форме (меню, поиск, прокрутка);
- проверка заполнения полей валидными данными;
- валидация сообщений об успешной отправке;
- написание отчётов

- сценарии с неотмеченными чек‑боксами;
- валидация сообщений об ошибках;
- оптимизация таймаутов и ожиданий
